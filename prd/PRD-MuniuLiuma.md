# MuniuLiuma（木牛流马）— 产品需求文档

> **版本**: v0.4-draft
> **日期**: 2026-07-22
> **状态**: Draft

---

## 1. 项目概述

### 1.1 背景

AI 编程助手（Claude Code、Cursor、Copilot 等）的能力越来越强，但存在一个核心问题：**AI 的执行力很强，但"理解需求"的能力很不稳定。**

具体表现：
- 给 AI 一段 PRD，它经常漏读关键约束
- AI 不知道"这个需求和之前的功能有什么关系"
- AI 设计出来的架构缺少全局一致性
- AI 生成的测试覆盖不全

业界已经出现了 Spec-Driven Development（SDD）方案（Liatrio SDD、Gentle-AI、OpenSpec、Spec-Kit），它们解决了 **"Spec → 代码 → 验证"** 的流程问题。但所有方案都假设**需求已经被想清楚了**——用户直接告诉 AI "做什么"。

**真正的缺口在更上游：原始 PRD / 需求文档 → 结构化 Spec。**

### 1.2 愿景

**MuniuLiuma（木牛流马）是一套纯 Skill 包，覆盖产品开发全流程，从原始需求到可交付代码。不依赖任何外部框架，开箱即用。**

### 1.3 名字由来

木牛流马是诸葛亮发明的运输工具，核心价值是**把物资从起点高效地运到终点**。MuniuLiuma 的核心价值是**把需求从原始 PRD 高效地转化为可交付的代码**。

- "流"字与"流程"双关，暗示全流程覆盖
- 拼音名 `muniu` 搜索唯一性极高，无重名

## 2. 市场调研与竞品分析

### 2.1 竞品分析

#### Liatrio SDD（Skill/Prompt 集合，无依赖）

- **核心流程**：4 步 SDD（Specification → Task Breakdown → Implementation → Validation）
- **关键能力**：
  - ✅ **范围校验**（Scope Validation）— 自动检测需求规模是否适合走全流程 → 已实现于 SKILL.md「范围评估」
  - ✅ **上下文腐化检测**（Context Verification Markers）— 每个 Phase 输出验证标记，检测上下文丢失 → 已实现于各 Phase reference
  - ✅ **Proof Artifacts** — 每个任务要求提供实现证据，验证时对照 → 已实现于 Phase 3 + Phase 5
  - ✘ **澄清问题独立文档** — 将模糊项单独成文档管理，未采纳（当前嵌入 Spec 中已可行）

#### Gentle-AI（Go CLI）

- **核心流程**：6 阶段 SDD（explore → propose → spec → design → implement → verify）
- **关键能力**：
  - ✅ **Engram 持久记忆** — 跨会话保持上下文和决策历史
  - ✅ **自动触发建议** — 根据任务复杂度自动建议是否走 SDD 流程 → 已实现于 SKILL.md「触发判断 > 第三步」
  - ✅ **委托停止规则**（Delegation Stop Rules）— 明确定义 Agent 何时应暂停并确认 → 已实现于 SKILL.md「Agent 停止规则」
  - ✘ **多模型支持** — 可切换不同 AI 模型，未采纳（宿主平台控制，非 Skill 层职责）

#### OpenSpec（npm CLI）

- **核心流程**：4 步（Proposal → Definition → Apply → Archive）
- **关键能力**：
  - ✅ **Delta 模式** — ADDED/MODIFIED/REMOVED 分类管理需求变更 → 已实现于 Phase 1「增量更新场景」
  - ✅ **Brownfield-first 策略** — 优先支持已有代码库的增量设计 → 已实现于 SKILL.md「项目模式识别」
  - ✘ **社区扩展生态** — 社区共建扩展库，未采纳（当前阶段无需，未来视需求再考虑）

#### GitHub Spec-Kit（Python CLI）

- **核心流程**：4 步 SDD（Spec → Plan → Tasks → Implement）
- **特点**：Constitution 作为项目级前置配置而非流程步骤；拥有 138 个社区扩展 + 25 个预设 + 35 个集成
- **借鉴**：当前阶段无需社区扩展生态

#### Kiro（AWS IDE）

- **核心流程**：Spec-Driven IDE，需求 → 用户故事 → 设计 → 任务 → 实现
- **关键能力**：
  - ✅ **自动化推理检查需求矛盾** — 系统化检查需求间的矛盾和缺口 → 已实现于 Phase 1「模糊检测 > 矛盾检测」
  - ✘ **Property-based Testing** — 基于属性约束生成测试用例，未采纳（超出范围，不写代码）

#### Spec Doc Generator（Claude Skill）

- **核心流程**：输入想法 → 输出需求文档/设计文档/实现计划三份文档
- **借鉴**：流程简单，无特别机制值得采纳

### 2.2 全流程缺口与 MuniuLiuma 定位

```
                     MuniuLiuma 覆盖的全流程
                     ┌──────────────────────────────┐
                     │                              │
  原始 PRD           │  解析 → Spec → 设计 → 任务 → 审计  │  可交付
  (Word/PDF/文字)    │                              │  (代码)
                     │  ↑ 现有方案全部从这里开始      │
                     └──────────────────────────────┘
```

**关键缺口**：没有任何现有方案覆盖"原始 PRD → 结构化 Spec"这个环节。所有方案都假设用户已经想清楚了需求。

**MuniuLiuma 的差异化定位**：
- 起点是用户丢来一份 PRD，而非用户说"我要做什么"
- 零依赖，不需要安装任何工具或配置项目文件
- 输出直接返回给用户，不写入项目目录
- 开箱即用，无需任何前置配置

## 3. 产品定位

### 3.1 目标用户

| 用户类型 | 场景 | 痛点 |
|---------|------|------|
| **产品经理** | 写完 PRD 后，需要让开发团队快速理解 | PRD 太长，开发看不完 |
| **开发工程师** | 拿到 PRD 后开始编码 | 不知道需求哪些地方模糊，边写边问 |
| **技术负责人** | 确保多个需求架构一致性 | 每个需求各自独立设计，架构碎片化 |
| **测试工程师** | 根据需求写测试用例 | 需求点遗漏，验收标准模糊 |

### 3.2 解决的问题

1. **PRD 太长，AI 读不完** → 结构化提取关键信息
2. **需求模糊点没人发现** → 自动生成追问清单
3. **架构设计各自为政** → 基于结构化 Spec 做一致设计
4. **实现和需求对不上** → 完工审计检查覆盖

### 3.3 不做什么

- 不写代码（交给 LLM 原生能力）
- 不创建项目文件（不侵入用户项目）
- 不依赖任何外部框架（纯 Skill）
- 不做 CI/CD 集成

## 4. Skill 包总览

### 4.1 架构

```
用户输入（PRD/需求文档/功能描述）
        │
        ▼
┌────── MuniuLiuma（单一 SKILL，5 个 Phase）──────────────┐
│                                                         │
│  Phase 1：prd-parser         原始文档 → 结构化 Spec + 追问   │
│                                                         │
│  Phase 2：arch-designer      结构化 Spec → 架构设计 + API    │
│                                                         │
│  Phase 3：task-planner       架构设计 → 可执行任务列表        │
│                                                         │
│  Phase 4：implementation     任务清单 → 实现指导规范          │
│                                                         │
│  ── Agent 按实现指导开发代码 ──                            │
│                                                         │
│  Phase 5：trace              全链路产物汇聚 → 完工审计报告    │
│                                                         │
└─────────────────────────────────────────────────────────┘
        │                           ▲
        │                           │
        ▼                           └── Agent 决策回溯补齐
完工审计报告
```

### 4.2 命令结构

单一 SKILL，通过子命令或自然语言指定入口 Phase：

```
/mnlm           全流程（Agent 根据意图自动裁剪执行范围）
```

> Agent 会根据输入和上下文状态自动补齐缺失的上游 Phase。例如用户说"做架构设计"但无 Spec → 先执行 Phase 1 再执行 Phase 2。

### 4.3 设计原则

MuniuLiuma 只做三件事：
1. **定义输入** — 每个 Phase 明确需要什么
2. **定义输出** — 每个 Phase 明确产出的格式和内容
3. **定义能力边界** — 输入缺失时告知 Agent 能做什么、不能做什么

**不做什么**：
- 不编排执行顺序（Agent 根据用户意图 + 上下文自行决策）
- 不管理跨 Phase 的状态（由对话上下文承载）
- 不同时涉及全部 Phase（每个调用只输出目标 Phase 的结果）
- 不包含自动化测试能力（测试是独立领域）
- **不替 Agent 做决策** — Phase 只报告"缺什么、后果是什么"，由 Agent 决定后续策略

**语言策略**：
- Skill 指令本身：使用中文（目标用户中文开发者为主）
- 输出产物（Spec / 设计文档等）：默认中文，可通过 `--lang=en` 切换英文

### 4.4 文件结构

单一 SKILL 目录，各 Phase 通过 reference 文件按需加载：

```
.trae/skills/muniu-liumia/
├── SKILL.md                        # 主控：触发判断 + 决策表 + 前置依赖检查
└── references/
    ├── phase-1-prd-parser.md       # Phase 1：需求解析（与 5.1 定义对应）
    ├── phase-2-arch-designer.md    # Phase 2：架构设计（与 5.2 定义对应）
    ├── phase-3-task-planner.md     # Phase 3：任务规划（与 5.3 定义对应）
    ├── phase-4-implementation.md   # Phase 4：实现指导（与 5.4 定义对应）
    └── phase-5-trace.md            # Phase 5：完工审计（与 5.5 定义对应）
```

Agent 只加载当前执行所需的 reference 文件，不额外消耗上下文。

### 4.5 非功能性需求

| 维度 | 要求 | 说明 |
|------|------|------|
| **性能** | 单次 Phase 输出应在 Agent 单次响应内完成 | 不拆分为多次对话轮次，避免用户等待 |
| **容错性** | 输入不规范时降级而非报错 | 如 PRD 格式混乱仍可尝试提取，无法提取时明确告知 |
| **一致性** | 同一输入多次执行 → 输出结构和覆盖度一致 | 不随 Agent 状态波动而遗漏功能点 |
| **可移植性** | 输出产物为纯 Markdown，不依赖特定渲染器 | 可粘贴到任何 Markdown 编辑器或项目管理工具 |
| **上下文友好** | 输出精简，只包含结构化结果 | 不包含分析过程、不输出备选方案完整论述 |

> **不设硬性 token 上限**：Skill 无法控制宿主平台的上下文窗口大小（128K / 200K / 1M 不等），硬性截断会导致长 PRD 解析不完整，违背核心原则。通过 4.7 的策略指令来控制不必要的消耗。

### 4.6 多平台适配

MuniuLiuma 的核心是纯 Markdown Skill 文件，通过调整目录结构适配不同平台：

| 平台 | Skill 目录 | 格式差异 |
|------|-----------|----------|
| **Trae** | `.trae/skills/muniu-liumia/` | SKILL.md + references/ |
| **Claude Code** | `.claude/skills/muniu-liumia/` | SKILL.md + references/（相同格式） |
| **Cursor** | `.cursor/rules/muniu-liumia/` | 适配 Cursor Rules 格式 |
| **其他平台** | 按平台 Skill/Rules 规范放置 | 核心内容不变，仅调整元数据格式 |

**适配原则**：
- Skill 指令内容（SKILL.md + 各 Phase reference）跨平台复用，不针对平台修改核心逻辑
- 仅适配目录结构和元数据格式（如触发命令前缀、文件命名规范）
- 各平台的适配版本保持功能对等

### 4.7 上下文管理策略

Skill 不设置硬性 token 上限（原因见 4.5），但通过以下指令策略控制不必要的上下文消耗：

1. **按需加载 reference**
   - 每次调用只加载目标 Phase 对应的 reference 文件
   - 不预加载下游 Phase 的 reference

2. **中间产物传递而非重复生成**
   - 上游 Phase 的输出作为中间产物保留在上下文中，下游 Phase 直接引用
   - 不要求下游 Phase 重新生成或复述上游产物

3. **大文档分段处理**
   - 当 PRD 超过一定长度时，prd-parser 按章节/模块分段处理，每段独立提取后合并，而非一次性加载全文

4. **输出精简原则**
   - 每个 Phase 的输出只包含结构化结果，不包含分析过程
   - 架构设计不输出备选方案的完整论述，只输出最终方案 + 简短选型理由

5. **产物外部化建议**
   - 每个 Phase 输出末尾建议用户将产物保存为文件
   - 用户在新对话中可通过引用文件恢复上下文，避免跨会话重复消耗

6. **上下文溢出降级**
   - 当 Agent 感知到上下文接近窗口极限时，优先保证目标 Phase 的输入和输出完整
   - 可裁剪的内容：分析过程、中间推理步骤、非关键的上下文产物
   - 不可裁剪的内容：目标 Phase 的必需输入和结构化输出

## 5. Phase 详细定义

### 5.1 Phase 1：prd-parser（需求解析）

**职责**：将原始 PRD / 需求文档 / 功能描述转化为结构化 Spec，**确保不遗漏任何需求细节、不放过任何模糊项。**

**输入**：
- 纯文本（粘贴或消息中的文字）
- 本地文件内容（读取后传入，如 `.md` / `.txt`）
- 网页内容（通过 WebFetch 抓取后传入）
- 无输入时主动追问获取信息
- **多份文档**：逐份处理，每份产出对应的独立 Spec

> **只接受文本输入**。文件读取、网页抓取、内容提取均在调用前完成。

**质量保障机制**：

```
┌─────────────────────────────────────────────────────┐
│                 输入预处理（Agent 负责）                 │
│                                                       │
│  文件/链接 → 提取文字内容                               │
│  无法提取 → 告知用户并提供替代方案                       │
│                            │                          │
│                            ▼                          │
│                 prd-parser 处理流程                     │
│                                                       │
│  原始文档（多份则逐份处理）                                 │
│     │                                                  │
│     ▼                                                  │
│  ┌─────────────┐                                       │
│  │ 第一遍：粗提取 │ → 功能点清单 + 约束条件（广度覆盖）      │
│  └──────┬──────┘                                       │
│         ▼                                              │
│  ┌─────────────┐                                       │
│  │ 第二遍：细校验 │ → 逐段对照原文，检查是否有遗漏         │
│  └──────┬──────┘                                       │
│         ▼                                              │
│  ┌─────────────┐                                       │
│  │ 第三遍：模糊检测│ → 标记每个功能点中的模糊表述           │
│  └──────┬──────┘                                       │
│         ▼                                              │
│  ┌─────────────┐                                       │
│  │ 输出         │ → 每份文档对应一份结构化 Spec + 追问清单 │
│  └─────────────┘                                       │
│                                                       │
└─────────────────────────────────────────────────────┘
```

**三遍校验法（核心机制）**：

| 遍数 | 目标 | 方法 | 输出 |
|------|------|------|------|
| **第一遍：粗提取** | 广度覆盖，不漏功能点 | 通读全文，识别所有功能点和约束 | 功能点清单 + 约束条件 |
| **第二遍：细校验** | 精度保证，不遗漏细节 | 逐段对照原文，查漏补缺 | 修正后的完整清单 |
| **第三遍：模糊检测** | 质量达标，不模棱两可 | 逐条检查每个功能点的表述是否明确 | 追问清单 |

**模糊检测的具体维度（第三遍）**：

| 模糊类型 | 示例原文 | 检测方式 | 生成的追问 |
|---------|---------|---------|-----------|
| **量化缺失** | "支持快速响应" | 无具体数字/指标 | "快速"的具体指标是？<1s？<500ms？ |
| **主语缺失** | "进行权限校验" | 谁来做？ | 是前端校验还是后端校验？ |
| **条件缺失** | "异常时给出提示" | 什么异常？什么提示？ | 具体异常类型？提示文案/弹窗？ |
| **程度模糊** | "兼容主流车型" | 哪些算主流？ | "主流车型"具体指哪些品牌和型号？ |
| **范围模糊** | "部分用户可见" | 哪些部分？ | 划分规则是什么？按角色/标签/权限？ |
| **时态模糊** | "后续版本支持" | 当前版本做不做？ | 这是当前版本的 P0 还是未来版本规划？ |
| **参照模糊** | "类似微信那样" | 像的是哪个功能？什么程度？ | 微信的哪个具体功能？交互逻辑/视觉风格/全部？ |
| **冲突未处理** | 多处说法不一致 | 交叉对比发现矛盾 | A 处说"必须登录"，B 处说"游客可用"，以哪个为准？ |

**输出**：

```markdown
# 结构化 Spec

## 功能点清单
| ID | 功能点 | 优先级 | 关联需求 |
|----|-------|--------|---------|
| F-01 | ... | P0 | ... |

## 验收标准
| ID | 功能点 | Given | When | Then |
|----|-------|-------|------|------|
| AC-01 | F-01 | ... | ... | ... |

## 约束条件
- 技术约束：...
- 业务约束：...
- 安全约束：...

## 模糊项
| 问题 | 建议 | 优先级 |
|------|------|--------|
| "..."未明确 | 建议确认... | 高/中/低 |

## 依赖关系
- 本功能依赖已有功能：...
- 本功能被其他功能依赖：...
```

**处理流程**：
1. 识别输入内容份数，多份文档则标记来源后逐份处理
2. **第一遍**：通读全文，识别并提取所有功能点（广度优先，宁可多列不漏列）
3. **第二遍**：逐段对照原文，检查第一遍的提取结果是否有遗漏或偏差
4. 为每个功能点生成验收标准（Given/When/Then）
5. 提取显式/隐式约束条件
6. **第三遍**：逐条进行模糊检测，标记所有模糊项
7. 分析功能间的依赖关系
8. 输出对应的结构化 Spec + 追问清单（多份文档则输出多份 Spec）

**硬性规则**：
- 每个功能点必须有至少一个验收标准
- **模糊项必须标记，不能默认"理解"** — 宁可多问不错过
- 必须区分 P0/P1/P2 优先级
- 不修改用户原始意图，只做结构化提取
- **第二遍校验未完成前，不能输出最终结果**
- **第三遍检测出模糊项但未生成追问，不能输出最终结果**

**追问回填机制**：

当用户回答追问清单中的问题后，Agent 应将答案回填到 Spec 中：

1. 用户回答追问项 → Agent 将答案回填到 Spec 对应的功能点描述中
2. 回填后更新验收标准（如追问答案影响了验收条件）
3. 将已回填的模糊项状态从“待确认”更新为“已确认”
4. 如追问答案引发新的模糊项 → 补充到追问清单中

> **触发方式**：用户回答追问时，Agent 自动触发回填，无需重新执行完整的 prd-parser 流程。
> 回填完成后输出更新摘要：“已回填 N 项，剩余 M 项待确认，新增 K 项追问。”

**增量更新场景**：

当用户提供已解析过的 PRD 的更新版本时：

1. 对比新旧版本 PRD 的差异（新增/修改/删除的内容）
2. 仅重新解析变更部分，更新 Spec 中对应的功能点、验收标准和约束条件
3. 保留未变更部分的原有 Spec 内容
4. 标注变更影响范围：“以下 N 个功能点因 PRD 更新而变更，下游产物（架构设计/任务清单）可能已过期”

> 增量更新仅在用户明确提供“更新版 PRD”时触发。普通的新 PRD 仍走全量解析流程。

### 5.2 arch-designer（架构设计）

**职责**：基于结构化 Spec 和项目现有架构上下文，输出一致、可落地的架构设计方案。聚焦当前需求，同时标识可扩展点以备未来演进。

**输入**：
- **[必需] 结构化 Spec** — 接受单个或多个 Spec（来自 prd-parser 输出，或用户直接提供）
  - 多个 Spec 时，Skill 会先判断它们之间的关系，再决定输出方式
- **[可选] 项目现有架构上下文** — Agent 在项目目录中自动收集
  - 技术栈（语言、框架、数据库、中间件）
  - 现有模块划分及职责
  - API / 接口风格
  - 关键数据流和依赖关系
  - 已有架构文档（如有）
- 用户可补充技术栈偏好

> **项目上下文由 Agent 负责收集**。当用户在项目目录中调用时，Agent 通过扫描项目结构、读取关键配置文件、分析现有代码等方式获取上下文信息。
> 当 Agent 无法获取项目上下文（如不在项目目录中调用）时，Skill 正常设计，输出末尾增加提示：“未检测到项目上下文，建议在项目目录中调用以获得更好的架构一致性。”

**项目上下文收集范围**：

| 收集项 | 来源示例 | 收集深度 |
|---------|---------|----------|
| 技术栈 | `package.json`、`pom.xml`、`go.mod`、`Cargo.toml` | 直接读取配置文件 |
| 模块划分 | 项目目录结构、`src/` 下的子目录 | 扫描目录层级（最深 3 层） |
| 现有 API/接口 | 路由文件、Controller 类、OpenAPI 定义 | 读取路由定义文件，不深入实现 |
| 已有架构文档 | `ARCHITECTURE.md`、`docs/` 目录 | 仅读取文档摘要 |
| 依赖关系 | 模块间 import/引用关系 | 分析顶层依赖，不做递归分析 |

> **收集策略**：优先读取配置文件和目录结构（成本低、信息密度高），代码分析仅作为补充。
> 当项目规模较大时，Agent 可主动询问用户“项目核心模块有哪些”以替代全量扫描。

> **前置依赖：结构化 Spec** — arch-designer 的必需输入是结构化 Spec。如果调用时 Spec 不可用（对话上下文中没有），Agent 应先检查用户是否提供了原始 PRD/需求文档：
> 1. 如果用户已提供原始文档 → Agent 先调用 prd-parser 生成 Spec，再将 Spec 传给 arch-designer
> 2. 如果用户也未提供原始文档 → Agent 主动询问用户"请提供需求文档或结构化 Spec"
> 3. **不允许在无 Spec 的情况下直接运行 arch-designer 进行推测性设计**

**处理流程**：

**步骤 A：交叉分析** — 先判断 Spec 间关系
1. 读入所有结构化 Spec，识别 Spec 数量和功能域
2. 分析 Spec 间的关系：
   a. **模块重叠检测** — 是否有多个 Spec 定义了相同或相似的模块
   b. **接口依赖检测** — 是否有 Spec 的输出是另一个 Spec 的输入
   c. **技术栈冲突检测** — 是否有 Spec 提出了不一致的技术选型
   d. **完全不相关** — 多个 Spec 分属不同的功能域，无交叉
3. 如果有项目上下文，同时分析新设计对现有架构的影响
4. 输出关系类型：强相关 / 弱相关 / 不相关

**步骤 B：分层设计** — 根据关系选择设计策略
1. 根据关系类型执行对应的设计策略：

   | 关系 | 策略 |
   |------|------|
   | **强相关**（共享模块、数据依赖） | 统一设计一套架构，所有 Spec 的需求一次覆盖 |
   | **弱相关**（少量接口交互） | 各自独立设计，但接口定义时对齐，标注跨 Spec 边界 |
   | **不相关**（独立功能域） | 完全独立设计，互不影响 |

2. 如果用户提供了技术栈偏好，优先使用
3. 标识可扩展点

**步骤 C：方案输出**
1. 根据关系类型选择输出形式：统一架构 / 标注边界 / 分开输出
2. 输出架构设计方案

**输出**：

```markdown
# 架构设计方案

## 多 Spec 关系统计
- 本次覆盖 Spec 数量：N 个
- 关系类型：强相关 / 弱相关 / 不相关
- 共享模块数量：N 个

## 技术选型
- 语言/框架：...
- 关键库：...
- 选型理由：...

## 模块划分
| 模块 | 职责 | 所属 Spec | 对外接口 | 新增/修改 |
|------|------|----------|---------|----------|
| ...  | ...  | Spec-01   | ...      | 新增      |

## 需求 → 模块映射
| Spec | 功能域 | 涉及模块 |
|------|-------|---------|
| Spec-01 | 语音唤醒 | 唤醒引擎、指令解析 |
| Spec-02 | 空调控制 | 指令解析、空调执行 |
| Spec-03 | 车窗控制 | 指令解析、车窗执行 |

## 数据流
- 请求流程：...
- 关键路径：...

## 接口定义
- API / 方法签名：...

## 架构影响分析
### 与已有架构的关系
- 新增模块 vs 已有模块
- 对已有模块的影响范围
- 是否需要修改已有模块

### 架构一致性检查
| 维度 | 检查结果 | 说明 |
|------|---------|------|
| 模块职责 | ✅ / ⚠️ / ❌ | 新模块与已有模块无职责重叠 |
| 接口风格 | ✅ / ⚠️ / ❌ | 新 API 与项目风格一致 |
| 技术栈 | ✅ / ⚠️ / ❌ | 无冲突的技术选型 |
| 数据流 | ✅ / ⚠️ / ❌ | 数据流集成合理 |

### 现有架构优化建议（如有）
- 检测到现有模块 X 的职责过重，建议拆分
- 检测到模块 Y 存在硬编码依赖，建议改为接口注入

## 技术风险评估
| 风险项 | 风险类型 | 影响程度 | 建议应对策略 |
|---------|---------|----------|------------|
| 依赖库 `XXX` 成熟度低 | 技术风险 | 高/中/低 | 预留替换接口，封装依赖 |
| 模块 `YYY` 实现复杂度超出团队经验 | 实现风险 | 高/中/低 | 建议先做技术验证 |
| 性能目标“响应 < 200ms”未经验证 | 性能风险 | 高/中/低 | 建议原型验证 |

## 架构边界与演进方向
### 当前覆盖范围
- 本次设计覆盖的功能域：...

### 可扩展点（标记位置，不做预实现）
- 接口 `XXX` 设计了通用格式，未来新增类型无需修改核心逻辑
- 模块 `YYY` 预留了插件位，当前只实现默认策略

### 领域演进提示（如有）
- 检测到项目所属领域（如车载语音），常见演进方向包括...
- 当前设计已兼容上述方向，但未实际实现
```

**输出方式根据关系类型不同**：

| 关系 | 输出方式 |
|------|---------|
| **强相关**（共享模块、数据依赖） | 一套统一架构方案，包含需求→模块映射表 |
| **弱相关**（少量接口交互） | 统一架构，标注模块边界和跨 Spec 接口 |
| **不相关**（独立功能域） | 分开的多套架构方案，各出一份，在同一个消息中返回 |

**硬性规则**：
- **有项目上下文时**：必须做一致性检查，不一致项不得跳过，必须在输出中标注
- **无项目上下文时**：必须提示用户"未检测到项目上下文"
- **多个 Spec 时**：必须先独立设计再交叉分析，不得跳过 Phase 1 直接合并
- **不相关 Spec**：必须分开输出，不得强行合并为统一架构
- 必须显式标注"新增、修改、不变"三类模块的关系
- 不设计实现细节（留给代码生成阶段）
- 如果检测到现有架构有明显不足，必须提出优化建议（不强制采纳，但需要标识）
- **不做未来需求的预实现** — 可扩展点只标记接口位和兼容性说明，不写入具体实现逻辑
- **有项目上下文时**：必须评估技术风险，高风险项不得跳过，必须在输出中标注应对策略

> **输出提示**：建议将架构设计方案保存为文件，作为项目架构文档供后续需求参考。

### 5.3 task-planner（任务规划）

**职责**：将架构设计方案拆解为可独立开发的任务列表。

**输入**：
- **[必需] 架构设计方案** — 来自 arch-designer 输出（模块划分、接口定义），或用户直接提供的设计描述

> **前置依赖：架构设计方案** — task-planner 的必需输入是架构设计方案。如果调用时架构设计不可用：
> 1. Agent 先检查对话上下文中是否有结构化 Spec
>    - 有 Spec → 先调用 arch-designer 生成架构设计，再传给 task-planner
>    - 无 Spec → 检查用户是否提供过原始 PRD，如有则按 prd-parser → arch-designer → task-planner 链路补齐
> 2. 如果原始 PRD 和 Spec 均不可用 → Agent 主动询问用户"请提供架构设计或需求文档"
> 3. **不允许跳过架构设计直接生成任务**（任务必须基于确定的模块划分和接口定义）

**输出**：

```markdown
# 任务列表

## 任务清单
| ID | 任务 | 所属模块 | 依赖 | 预估工作量 | 验收条件 |
|----|------|---------|------|-----------|---------|
| T-01 | 实现用户注册 API | 用户服务 | — | 中等（2h） | POST /register 返回 201，密码已加密 |
| T-02 | 实现用户登录 API | 用户服务 | T-01 | 中等（1.5h） | POST /login 返回 JWT Token |
| T-03 | 实现密码重置 | 用户服务 | T-01 | 复杂（3h） | 发送重置邮件，重设密码生效 |
| T-04 | 实现通知推送接口 | 通知服务 | — | 简单（1h） | 推送消息写入队列 |

## 执行顺序
- 串行依赖：T-01 → T-02 → T-03
- 可并行：T-01 和 T-04 可并行开发
```

**处理流程**：

1. **识别架构层次** — 按 UI 层 → 业务逻辑层 → 数据访问层 → 基础设施层 逐层提取模块，层内按组件/功能进一步细分。
2. **按拆分规则生成任务** — 遵循以下规则拆解：

   **拆分依据（按优先级）**：
   - **按架构层次**：每一层作为一个任务分组，层内各组件分别独立成任务。
   - **按接口/数据流**：每个 API 端点、每条数据流处理路径独立为一个任务。
   - **按功能点**：一个完整功能的最小可验证单元（如"用户注册"、"密码重置"）独立为任务。

   **粒度标准**：
   - 理想任务规模：**独立可开发、可测试、1-2 小时内完成**，最长不超过 **4 小时**。
   - 超 4 小时的任务须进一步拆解；不足 30 分钟的任务合并到相邻任务中。

   **依赖识别规则**：
   - 被依赖模块（数据模型、工具函数、基类等）优先拆出，排入早期任务。
   - 同一层次内无相互引用的组件标记为 **可并行执行**。
   - 依赖链长度不超过 **3 层**（A → B → C），超过则合并中间任务。

   **工作量估算标准**：
   - 简单（≤1h）：修改变量名、配置文件、单一工具函数。
   - 中等（1-2h）：新增一个 CRUD 接口、一个无交互的 UI 组件。
   - 复杂（2-4h）：新增一个完整页面含交互、跨层联调功能。

   **验收条件要求**：
   - 必须为 **可客观验证** 的描述。例如✅ "用户注册接口返回 201 且密码已加密存储"，而非 ❌ "注册功能正常"。

3. **识别任务间依赖关系** — 根据上述依赖规则自动推断并生成依赖图。
4. **标记可并行执行的任务** — 同一层次无依赖关系的任务归入同一并行批次。
5. **为每个任务指定验收条件** — 按验收条件要求编写可验证断言。

> **输出提示**：建议将任务列表保存为文件，以接入项目管理工具或供后续会话使用。

**任务状态跟踪**：

Skill 不管理任务状态（由 Agent 或外部项目管理工具负责），但任务清单的输出格式支持状态追踪：

- 每个任务 ID 可附加状态标记：`待开发` / `开发中` / `已完成` / `已验证`
- 用户再次要求"更新任务"时，如上下文中已有任务状态信息，Skill 仅更新未完成的任务，已完成的保留原样
- 任务清单可导出为常见项目管理工具可识别的格式（Markdown 表格 / 简单 CSV）

**任务变更与回退**：

当某个任务在开发中发现不可行（技术方案无法实现、依赖不满足等），Agent 可请求 task-planner 重新规划：

1. 标记失败任务及原因
2. 生成替代任务或调整方案
3. 重新评估受影响任务的依赖链和执行顺序
4. 标注变更原因和影响范围：“任务 T-XX 因 XX 原因调整，影响下游任务 T-YY、T-ZZ”

### 5.4 Phase 4：implementation（实现指导）

**职责**：将任务清单转化为结构化的实现指导规范，为 Agent（或人类开发者）提供编码阶段的行为约束、执行策略和质量基准。

**输入**：
- **[必需] 任务清单** — 任务 ID、依赖关系、验收条件（task-planner 输出）
- **[必需] 架构设计方案** — 模块划分、接口定义、技术选型（arch-designer 输出）
- **[可选] 结构化 Spec** — 功能点、约束条件（用于生成一致性测试要求）

**前置依赖**：不允许在无任务清单的情况下生成实现指导。任务清单缺失时，先检查有无架构设计 → 有则先跑 Phase 3 → 无则继续回溯至 Phase 2 / Phase 1。

**输出内容**：

实现指导包含以下三个核心组件：

#### 1. 三层边界约束

为每个任务或任务组定义 Agent 的行为边界：

| 层级 | 含义 | 示例 |
|------|------|------|
| ✅ Always | 无需询问即可做 | 运行测试、遵循命名规范、记录日志 |
| ⚠️ Ask First | 需人类确认 | 修改数据库 Schema、添加新依赖、变更 API 签名 |
| 🚫 Never | 绝对禁止 | 提交密钥、修改 CI 配置、删除测试用例 |

#### 2. 分阶段执行策略

将任务按开发阶段分组，每个阶段完成后设置审查门禁：

```
Setup 阶段 → 审查门禁 → Core 阶段 → 审查门禁 → Advanced 阶段 → 审查门禁 → Testing 阶段
```

- 每个阶段内的任务可并行执行（无依赖关系的任务）
- 阶段间串行执行，前一阶段审查通过后才进入下一阶段
- 每个审查门禁检查：阶段内任务是否全部完成、验收条件是否达标、证据是否提供

#### 3. 一致性测试要求（Conformance Criteria）

从 Spec 的验收标准和架构设计的接口定义，派生出语言无关的测试套件：

- 每个功能点的 Given/When/Then 转化为具体的测试断言
- 每个 API 接口定义转化为输入/输出约束
- 性能约束转化为基准测试要求
- 安全约束转化为安全检查项

**输出格式**：

```markdown
# 实现指导规范

## 三层边界约束
| 层级 | 规则 |
|------|------|
| ✅ Always | ... |
| ⚠️ Ask First | ... |
| 🚫 Never | ... |

## 分阶段执行计划
### 阶段 1：Setup
- [ ] T-01: ... （验收条件 / 证据要求）
- [ ] T-02: ...
→ 审查门禁：检查项 1, 2, 3

### 阶段 2：Core
- [ ] T-03: ...
→ 审查门禁：检查项 1, 2, 3

## 一致性测试要求
| 来源 | 测试断言 | 类型 |
|------|---------|------|
| AC-01 | POST /register 返回 201 | 功能 |
| C-03 | 响应时间 < 200ms | 性能 |
| S-01 | 密码不可明文存储 | 安全 |
```

### 5.5 Phase 5：trace（完工审计）

**职责**：汇聚开发全流程全部产物，做完工审计，判断需求是否全部完成，输出是否有遗漏或未达标的交付缺口。

**输入**：
- **[必需] 结构化 Spec** — 功能点清单、验收条件（prd-parser 输出）
- **[必需] 架构设计方案** — 模块划分（arch-designer 输出）
- **[必需] 任务清单** — 任务 ID 和依赖关系（task-planner 输出）
- **[必需] 实现指导** — 三层边界约束、一致性测试要求（implementation 输出）
- **[必需] 实现代码** — 项目代码（用于代码扫描和审计）

> **前置依赖处理** — trace 需要全链路产物才能完成审计。当部分输入缺失时，按以下优先级补齐：
> 1. **检查对话上下文** — Agent 优先从当前对话中查找已有产物（Spec / 设计 / 任务清单 / 实现指导）
> 2. **缺失时自动补齐** — 对于缺失的上游产物，Agent 先补齐再审计：
>    - 缺失 Spec + 用户有原始 PRD → 调用 prd-parser
>    - 缺失架构设计 + Spec 可用 → 调用 arch-designer
>    - 缺失任务清单 + 架构设计可用 → 调用 task-planner
>    - 缺失实现指导 + 任务清单可用 → 调用 implementation
> 3. **用户提供原始来源** — 如果对话上下文和补齐链路均不可行（如原始 PRD 从未提供过），Agent 询问用户补充缺失的产物
> 4. **部分审计** — 如果用户明确要求"只看能查的部分"，Agent 可根据已有产物跳过缺失维度做部分审计，并在报告中标注"XX 维度因缺少 XX 输入未审计"

**处理流程**：

1. **收集输入** — 从对话上下文或用户提供方式获取全链路产物，按上述前置依赖规则补齐
2. **执行审计** — 从五个维度审计完整性：

   | 审计维度 | 检查内容 | 数据来源 |
   |---------|---------|--------|
   | **功能覆盖** | Spec 中每个功能点是否有对应的任务记录或代码实现 | Spec × 任务清单 × 代码 |
   | **架构落地** | 架构设计中定义的每个模块是否在代码中有对应实现 | 架构设计 × 代码 |
   | **无主代码** | 是否有实现了但不对应任何 Spec 功能点的代码 | 代码 × Spec（反向扫描） |
   | **任务闭环** | 全部任务是否在代码中有对应实现 | 任务清单 × 代码 |
   | **验收标准达标** | Spec 中定义的 Given/When/Then 场景是否在代码逻辑中被满足 | Spec 验收标准 × 代码逻辑 |
   | **实现指导对照** | 代码实现是否遵循三层边界约束、一致性测试要求是否达标 | 实现指导 × 代码 |

3. **判定交付状态** — 根据审计结果输出总体判断：

**代码扫描判定标准**：

| 判定方式 | 适用场景 | 说明 |
|---------|---------|------|
| 文件/目录存在性检查 | 模块级审计 | 检查架构定义的模块是否在代码中有对应目录/文件 |
| 函数/类名匹配 | 任务级审计 | 检查任务描述的功能点是否有对应函数或类（精确匹配 + 模糊匹配） |
| 调用链追踪 | 功能覆盖审计 | 从入口点追踪调用链，检查完整功能路径是否实现 |

- **动态语言**（Python/JS/TS）：侧重运行时调用分析和 import/require 分析
- **静态语言**（Java/Go/Rust）：利用类型系统和编译时检查
- 允许模糊匹配：函数名不完全一致但语义相近时标记为“疑似已实现”，由用户确认

> **误判处理**：代码扫描可能产生误判（如函数存在但逻辑不完整），审计报告中对每个判定结果标注置信度（✅ 确认 / ⚠️ 疑似 / ❌ 未实现）。

- ✅ **可交付**：全部功能点已实现，架构模块已落地，无无主代码
- ⚠️ **有条件交付**：P0 全部实现，低优先级有缺口
- ❌ **不可交付**：存在 P0 未实现

**输出**：

## 6. 执行决策

MuniuLiuma 是一个单一 SKILL，内部按 Phase 1→5 顺序依赖。Agent 根据用户意图和上下文状态，自行决定执行哪些 Phase 以及执行顺序。

### 6.1 角色划分

| 角色 | 职责 |
|------|------|
| **Phase（即 SKILL 内部步骤）** | 定义输入、输出、能力边界。输入缺失时告知 Agent 缺什么、后果是什么，**但不替 Agent 做决策** |
| **Agent（大模型）** | 根据用户意图 + 上下文状态决定执行范围，回溯补齐缺失的上游产物，与用户沟通协商 |

### 6.2 决策表

Agent 收到用户指令后，按以下规则判断执行范围：

**第一步：确定目标 Phase**

| 用户意图 | 输出目标 |
|---------|----------|
| `/mnlm` 或"把这个需求做完" | 全链路走完（Phase 1→5） |
| "解析需求/分析文档" | Phase 1：结构化 Spec |
| "做架构设计" | Phase 2：架构设计方案 |
| "拆任务" | Phase 3：任务清单 |
| "生成实现指导" / "准备开发" | Phase 4：实现指导规范 |
| "检查需求完成情况" / "审计一下" | Phase 5：完工审计报告 |

**第二步：递归检查前置依赖**

Agent 确认目标 Phase 后，逐层检查其前置输入是否在对话上下文中可用：

```
目标 Phase N 的 [必需] 输入都存在？
  ├── ✅ 全部存在 → 直接从 Phase N 开始执行
  └── ❌ 缺失 → 回溯到 Phase N-1（缺失的输入正是 Phase N-1 的输出）
        ├── Phase N-1 的输入也存在 → 执行 Phase N-1 → 输出传给 Phase N
        └── 仍缺失 → 继续回溯...直至 Phase 1
```

**第三步：执行并输出**

- 按补齐后的链路依次执行，中间产物保留在对话上下文中
- 最终只输出目标 Phase 的结果
- 如果上游 Phase 在本次调用中重新执行过，标注"下游产物可能已过期"

### 6.3 执行示例

| 用户说 | 上下文状态 | Agent 决策 | 实际执行 |
|--------|-----------|-----------|---------|
| `帮我根据这个 PRD 做架构` | 有 PRD，无 Spec | Phase 2 缺输入 → 回溯 Phase 1 | Phase 1 → Phase 2 |
| "解析这个需求" | 无上下文 | Phase 1 有文档输入 → 直接执行 | Phase 1 |
| "审计一下" | 全链路产物齐全 | Phase 5 输入齐全 → 直接审计 | Phase 5 |
| "拆任务" | 仅有 Spec，无设计 | Phase 3 缺设计 → 回溯 Phase 2 | Phase 2 → Phase 3 |
| "准备开发" | 有任务清单 | Phase 4 有输入 → 直接生成实现指导 | Phase 4 |
| "审计一下" | 仅有代码 | Phase 5 多项输入缺失，回溯至 Phase 1 也缺原始 PRD | 告知 Agent 缺少需求基线，Agent 自行决策 |

### 6.4 错误处理与降级策略

当 Phase 执行过程中遇到异常时，Skill 应指导 Agent 采取降级而非中断的策略：

| 异常场景 | 处理策略 |
|---------|----------|
| Agent 无法读取项目代码（权限问题/非项目目录） | 跳过代码扫描环节，仅基于已有产物做部分审计，标注“代码审计未执行” |
| WebFetch 抓取链接失败 | 告知用户抓取失败，请求用户手动粘贴文档内容或导出文件 |
| LLM 推理能力不足导致 Spec 质量低 | 在输出中标注置信度低的条目，建议用户人工审核关键功能点 |
| 上下文溢出导致中间产物丢失 | 建议用户将已完成的产物保存为文件，在新对话中通过文件引用继续流程 |
| 输入文档格式异常（非文本、乱码等） | 告知用户支持的输入格式，请求提供可解析的版本 |

**降级原则**：
- 每个 Phase 即使降级也要输出**部分结果**，而非直接失败
- 降级输出必须明确标注哪些环节被跳过或简化，以及原因
- 始终为用户提供可操作的下一步建议

## 8. 成功指标

| 指标 | 目标 | 评测方法 |
|------|------|----------|
| PRD → Spec 提取准确率 | ≥ 90% 功能点被正确识别 | 准备 5-10 份不同领域的 PRD 样本，人工标注功能点基线，对比 Skill 输出的召回率 |
| 架构一致性 | 同项目多次设计 → 架构模式一致 | 同一项目输入 2-3 个不同需求，检查输出的架构设计是否遵循相同的设计模式和技术栈 |
| 用户追问率 | Spec 中的模糊项 ≥ 80% 被识别并生成追问 | 在 PRD 样本中预埋已知模糊点，统计 Skill 检测出的比例 |
| 完工审计覆盖 | Phase 5 审计覆盖全部 6 个维度（功能覆盖/架构落地/无主代码/任务闭环/验收标准达标/实现指导对照） | 使用已完成的小型项目，检查审计报告的维度完整性和判定准确性 |
| 用户满意度 | 实际用户认为产物有参考价值 | 收集早期用户的使用反馈，统计产物被实际采纳的比例 |
