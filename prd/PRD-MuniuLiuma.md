# MuniuLiuma（木牛流马）— 产品需求文档

> **版本**: v0.1-draft
> **日期**: 2026-07-13
> **状态**: Draft

---

## 1. 项目概述

### 1.1 背景

AI 编程助手（Claude Code、Cursor、Copilot 等）的能力越来越强，但存在一个核心问题：**AI 的执行力很强，但"理解需求"的能力很不稳定。**

具体表现：
- 给 AI 一段 PRD，它经常漏读关键约束
- AI 不知道"这个需求和之前的功能有什么关系"
- AI 设计出来的架构缺少全局一致性
- AI 生成的测试覆盖不全

业界已经出现了 Spec-Driven Development（SDD）方案（Liatrio SDD、Gentle-AI、OpenSpec、Spec-Kit），它们解决了 **"Spec → 代码 → 验证"** 的流程问题。但所有方案都假设**需求已经被想清楚了**——用户直接告诉 AI "做什么"。

**真正的缺口在更上游：原始 PRD / 需求文档 → 结构化 Spec。**

### 1.2 愿景

**MuniuLiuma（木牛流马）是一套纯 Skill 包，覆盖产品开发全流程，从原始需求到可交付代码。不依赖任何外部框架，开箱即用。**

### 1.3 名字由来

木牛流马是诸葛亮发明的运输工具，核心价值是**把物资从起点高效地运到终点**。MuniuLiuma 的核心价值是**把需求从原始 PRD 高效地转化为可交付的代码**。

- "流"字与"流程"双关，暗示全流程覆盖
- 拼音名 `muniu` 搜索唯一性极高，无重名

## 2. 市场调研与竞品分析

### 2.1 调研项目一览

| 项目 | 类型 | 核心能力 | 覆盖环节 | 依赖 |
|------|------|---------|---------|------|
| **Liatrio SDD** | Prompt 集合 | 4 步 SDD 流程：生成 Spec → 拆 Task → 执行 → 验证 | Spec → 代码 → 验证 | 无，纯 Prompt |
| **Gentle-AI** | Go CLI 工具 | 10 阶段 SDD 流程 + 持久记忆（Engram）+ 15+ Agent 配置 | 上下文初始化 → 探索 → 提案 → Spec → 设计 → 任务 → 执行 → 验证 → 归档 | 需安装 CLI |
| **OpenSpec** | npm CLI | 提案 → Spec → 设计 → 任务 → 执行 → 归档（Living Spec） | 提案 → 归档全流程 | 需安装 npm 包 |
| **GitHub Spec-Kit** | Python CLI | Constitution → Specify → Plan → Tasks → Implement | 从原则定义到执行 | 需安装 Python + uv |
| **Specification Doc Generator** | Claude Skill | 输入想法 → 输出需求文档/设计文档/实现计划 | 生成三份文档 | Claude Code |

### 2.2 环节覆盖对比

```
                     MuniuLiuma 覆盖的全流程
                     ┌──────────────────────────────┐
                     │                              │
  原始 PRD           │  Spec → 设计 → 任务 → 验证    │  可交付
  (Word/PDF/文字)    │                              │  (代码+用例)
                     │  ↑ 现有方案全部从这里开始      │
                     │  (Liatrio/Gentle-AI/          │
                     │   OpenSpec/Spec-Kit)          │
                     └──────────────────────────────┘
```

**关键缺口**：没有任何现有方案覆盖"原始 PRD → 结构化 Spec"这个环节。所有方案都假设用户已经想清楚了需求。

### 2.3 差异化定位

| 维度 | Liatrio SDD / Gentle-AI / OpenSpec | MuniuLiuma |
|------|-----------------------------------|------------|
| 起点 | 用户说"我要做什么" | 用户丢来一份 PRD |
| 终点 | 代码/验证通过 | 代码/验证通过 |
| 项目依赖 | 需要项目文件（AGENTS.md / openspec/） | **零依赖** |
| 输出方式 | 写入项目目录 | **直接返回给用户** |
| 使用条件 | 需安装 CLI / 配置项目 | **开箱即用** |

## 3. 产品定位

### 3.1 目标用户

| 用户类型 | 场景 | 痛点 |
|---------|------|------|
| **产品经理** | 写完 PRD 后，需要让开发团队快速理解 | PRD 太长，开发看不完 |
| **开发工程师** | 拿到 PRD 后开始编码 | 不知道需求哪些地方模糊，边写边问 |
| **技术负责人** | 确保多个需求架构一致性 | 每个需求各自独立设计，架构碎片化 |
| **测试工程师** | 根据需求写测试用例 | 需求点遗漏，验收标准模糊 |

### 3.2 解决的问题

1. **PRD 太长，AI 读不完** → 结构化提取关键信息
2. **需求模糊点没人发现** → 自动生成追问清单
3. **架构设计各自为政** → 基于结构化 Spec 做一致设计
4. **测试覆盖不全** → 从 Spec 生成测试矩阵
5. **实现和需求对不上** → 自动验证对齐

### 3.3 不做什么

- 不写代码（交给 LLM 原生能力）
- 不创建项目文件（不侵入用户项目）
- 不依赖任何外部框架（纯 Skill）
- 不做 CI/CD 集成

### 3.4 原始文档来源要求

Skill 本身不做平台 API 集成，但 AI Agent 具备网页抓取能力。输入来源按以下优先级处理：

| 文档来源 | 提供方式 |
|---------|---------|
| **可公开访问的链接**（飞书/Confluence/Google Docs 等分享链接） | 用户提供链接 → Agent 通过 `WebFetch` 自动抓取内容 |
| **需要登录的在线文档**（企业内网、权限受限的文档） | 用户复制全文粘贴，或导出为 Markdown 后提供 |
| **本地文件**（Word/PDF/Markdown） | 直接拖入 IDE 或引用 `file:///` 路径 |
| **纯文本/聊天记录** | 直接粘贴 |
| **无输入** | Skill 会主动追问引导用户提供信息 |

> **原则**：优先走 Agent 的网页抓取能力（用户只需提供链接），抓取失败时回退到用户手动粘贴。Skill 自身不绑定任何平台的 API，保持零依赖。

## 4. Skill 包总览

### 4.1 架构

```
用户输入（PRD/需求文档/功能描述）
        │
        ▼
┌─────────────────────────────────────────────────┐
│                 MuniuLiuma  Skill 包              │
│                                                   │
│  prd-parser     原始文档 → 结构化 Spec + 追问清单   │
│                                                   │
│  arch-designer  结构化 Spec → 架构设计 + API 定义   │
│                                                   │
│  task-planner   架构设计 → 可执行任务列表            │
│                                                   │
│  test-planner   结构化 Spec → 测试矩阵 + 用例模板   │
│                                                   │
│  verify         代码 vs Spec → 验证报告             │
│                                                   │
│  trace          需求 → 代码 → 实现追踪图谱           │
└─────────────────────────────────────────────────┘
        │
        ▼
输出结果（直接返回给用户，不写项目文件）
```

### 4.2 命令结构

每个 Skill 对应一个 slash command，使用 `/ml` 作为统一前缀：

```
/ml.parse    需求解析（prd-parser）
/ml.design   架构设计（arch-designer）
/ml.plan     任务规划（task-planner）
/ml.test     测试规划（test-planner）
/ml.verify   验证对齐（verify）
/ml.trace    需求追溯（trace）
```

命令格式：

```
/ml.<cmd> [input]

input:
  - 直接粘贴的文本/PRD
  - @文件引用（file:/// 或 IDE 内引用）
  - 无参数（Skill 会主动询问）
```

> **Skill 不涉及流程编排**。用户和 Agent 自行决定调用哪些 Skill 以及调用顺序。
> 每个 Skill 只定义自己的输入要求和输出格式，Agent 根据用户意图组合使用。

### 4.3 设计原则

MuniuLiuma 的 Skill 只做三件事：
1. **定义输入** — 明确需要什么（用户/Agent 负责提供）
2. **定义输出** — 明确产出的格式和内容
3. **输入不足时追问** — 缺信息就引导用户补充

**不做什么**：
- 不编排 Skill 的执行顺序（由 Agent 根据用户意图决定）
- 不管理跨 Skill 的状态（由对话上下文承载）
- 不提供全流程命令（用户和 Agent 自行组合）

## 5. Skill 详细定义

### 5.1 prd-parser（需求解析）

**职责**：将原始 PRD / 需求文档 / 功能描述转化为结构化 Spec，**确保不遗漏任何需求细节、不放过任何模糊项。**

**激活机制（双层判断）**：

**第一层：开发意图检测** — 用户是否表达了基于这份文档"做开发动作"的意图

| 意图类型 | 示例 | 是否触发 |
|---------|------|---------|
| **显式命令** | `/ml.parse`、`/ml.parse <文档>` | ✅ 触发 |
| **解析请求** | "分析/解析/拆解/整理这份需求"、"把这份文档转成 Spec" | ✅ 触发 |
| **开发动作前置** | "根据这个需求做架构设计"、"基于这个PRD排任务" | ✅ 触发（解析是隐含前置） |
| **质量审查** | "检查这份PRD有没有模糊点"、"需求有什么遗漏" | ✅ 触发 |
| **仅分享无动作** | "你看下这份需求"、"发你一个文档"、"先看看" | ❌ 不触发 |
| **无附加说明** | 直接甩一个链接/文件，无任何附加文字 | ❌ 不触发 |
| **闲聊评估** | "这份需求写得挺有意思"、"这PRD写得太长了" | ❌ 不触发 |

> 不触发场景下 Skill 保持静默，等待用户进一步指示。

**第二层：输入内容验证** — 用户发来的文档是否确实是"需求文档"

触发后，Skill 首先判断文档类型：
- **是 PRD / 需求文档 / 功能描述** → 进入解析流程
- **是其他类型文档**（API 文档、设计稿、会议纪要、周报、技术方案等） → 告知用户"这份文档看起来不是需求文档（检测到类型：XX）"，询问用户意图
- **无法判断** → 主动询问"这份文档是需求文档吗？需要我帮您解析吗？"

**输入**：
- 纯文本（直接粘贴或写在消息中）
- 本地文件路径（Agent 读取后传入，如 `.md` / `.txt` / `.docx` 等）
- 链接（Agent 通过 `WebFetch` 抓取后传入）
- 无输入（Skill 会主动追问获取信息）
- 来源不限（飞书/Confluence/Google Docs 等）
- **多份文档**：可一次提供多份，Skill 会逐份处理，每份产出对应的独立 Spec

> **Skill 只接受文本输入**。文件读取、网页抓取、内容提取均由 Agent 在调用 Skill 前完成。
> Agent 无法读取的文件格式应向用户说明并请求提供文字版本。

**质量保障机制**：

```
┌─────────────────────────────────────────────────────┐
│                 输入预处理（Agent 负责）                 │
│                                                       │
│  文件/链接 → 提取文字内容                               │
│  无法提取 → 告知用户并提供替代方案                       │
│                            │                          │
│                            ▼                          │
│                 prd-parser 处理流程                     │
│                                                       │
│  原始文档（多份则逐份处理）                                 │
│     │                                                  │
│     ▼                                                  │
│  ┌─────────────┐                                       │
│  │ 第一遍：粗提取 │ → 功能点清单 + 约束条件（广度覆盖）      │
│  └──────┬──────┘                                       │
│         ▼                                              │
│  ┌─────────────┐                                       │
│  │ 第二遍：细校验 │ → 逐段对照原文，检查是否有遗漏         │
│  └──────┬──────┘                                       │
│         ▼                                              │
│  ┌─────────────┐                                       │
│  │ 第三遍：模糊检测│ → 标记每个功能点中的模糊表述           │
│  └──────┬──────┘                                       │
│         ▼                                              │
│  ┌─────────────┐                                       │
│  │ 输出         │ → 每份文档对应一份结构化 Spec + 追问清单 │
│  └─────────────┘                                       │
│                                                       │
└─────────────────────────────────────────────────────┘
```

**三遍校验法（核心机制）**：

| 遍数 | 目标 | 方法 | 输出 |
|------|------|------|------|
| **第一遍：粗提取** | 广度覆盖，不漏功能点 | 通读全文，识别所有功能点和约束 | 功能点清单 + 约束条件 |
| **第二遍：细校验** | 精度保证，不遗漏细节 | 逐段对照原文，查漏补缺 | 修正后的完整清单 |
| **第三遍：模糊检测** | 质量达标，不模棱两可 | 逐条检查每个功能点的表述是否明确 | 追问清单 |

**模糊检测的具体维度（第三遍）**：

| 模糊类型 | 示例原文 | 检测方式 | 生成的追问 |
|---------|---------|---------|-----------|
| **量化缺失** | "支持快速响应" | 无具体数字/指标 | "快速"的具体指标是？<1s？<500ms？ |
| **主语缺失** | "进行权限校验" | 谁来做？ | 是前端校验还是后端校验？ |
| **条件缺失** | "异常时给出提示" | 什么异常？什么提示？ | 具体异常类型？提示文案/弹窗？ |
| **程度模糊** | "兼容主流车型" | 哪些算主流？ | "主流车型"具体指哪些品牌和型号？ |
| **范围模糊** | "部分用户可见" | 哪些部分？ | 划分规则是什么？按角色/标签/权限？ |
| **时态模糊** | "后续版本支持" | 当前版本做不做？ | 这是当前版本的 P0 还是未来版本规划？ |
| **参照模糊** | "类似微信那样" | 像的是哪个功能？什么程度？ | 微信的哪个具体功能？交互逻辑/视觉风格/全部？ |
| **冲突未处理** | 多处说法不一致 | 交叉对比发现矛盾 | A 处说"必须登录"，B 处说"游客可用"，以哪个为准？ |

**输出**：

```markdown
# 结构化 Spec

## 功能点清单
| ID | 功能点 | 优先级 | 关联需求 |
|----|-------|--------|---------|
| F-01 | ... | P0 | ... |

## 验收标准
| ID | 功能点 | Given | When | Then |
|----|-------|-------|------|------|
| AC-01 | F-01 | ... | ... | ... |

## 约束条件
- 技术约束：...
- 业务约束：...
- 安全约束：...

## 模糊项
| 问题 | 建议 | 优先级 |
|------|------|--------|
| "..."未明确 | 建议确认... | 高/中/低 |

## 依赖关系
- 本功能依赖已有功能：...
- 本功能被其他功能依赖：...
```

**处理流程**：
1. 识别输入内容份数，多份文档则标记来源后逐份处理
2. **第一遍**：通读全文，识别并提取所有功能点（广度优先，宁可多列不漏列）
3. **第二遍**：逐段对照原文，检查第一遍的提取结果是否有遗漏或偏差
4. 为每个功能点生成验收标准（Given/When/Then）
5. 提取显式/隐式约束条件
6. **第三遍**：逐条进行模糊检测，标记所有模糊项
7. 分析功能间的依赖关系
8. 输出对应的结构化 Spec + 追问清单（多份文档则输出多份 Spec）

**硬性规则**：
- 每个功能点必须有至少一个验收标准
- **模糊项必须标记，不能默认"理解"** — 宁可多问不错过
- 必须区分 P0/P1/P2 优先级
- 不修改用户原始意图，只做结构化提取
- **第二遍校验未完成前，不能输出最终结果**
- **第三遍检测出模糊项但未生成追问，不能输出最终结果**

### 5.2 arch-designer（架构设计）

**职责**：基于结构化 Spec 和项目现有架构上下文，输出一致、可落地的架构设计方案。聚焦当前需求，同时标识可扩展点以备未来演进。

**触发条件**：
- 用户拿到 Spec 后要求"做架构设计"
- 用户说"/ml.design"

**输入**：
- **[必需] 结构化 Spec** — 接受单个或多个 Spec（来自 prd-parser 输出，或用户直接提供）
  - 多个 Spec 时，Skill 会先判断它们之间的关系，再决定输出方式
- **[可选] 项目现有架构上下文** — Agent 在项目目录中自动收集
  - 技术栈（语言、框架、数据库、中间件）
  - 现有模块划分及职责
  - API / 接口风格
  - 关键数据流和依赖关系
  - 已有架构文档（如有）
- 用户可补充技术栈偏好

> **项目上下文由 Agent 负责收集**。当用户在项目目录中调用 `/ml.design` 时，Agent 通过扫描项目结构、读取关键配置文件、分析现有代码等方式获取上下文信息。
> 当 Agent 无法获取项目上下文（如不在项目目录中调用）时，Skill 正常设计，输出末尾增加提示："未检测到项目上下文，建议在项目目录中调用以获得更好的架构一致性。"

**处理流程**：

**Phase 1：交叉分析** — 先判断 Spec 间关系
1. 读入所有结构化 Spec，识别 Spec 数量和功能域
2. 分析 Spec 间的关系：
   a. **模块重叠检测** — 是否有多个 Spec 定义了相同或相似的模块
   b. **接口依赖检测** — 是否有 Spec 的输出是另一个 Spec 的输入
   c. **技术栈冲突检测** — 是否有 Spec 提出了不一致的技术选型
   d. **完全不相关** — 多个 Spec 分属不同的功能域，无交叉
3. 如果有项目上下文，同时分析新设计对现有架构的影响
4. 输出关系类型：强相关 / 弱相关 / 不相关

**Phase 2：分层设计** — 根据关系选择设计策略
1. 根据关系类型执行对应的设计策略：

   | 关系 | 策略 |
   |------|------|
   | **强相关**（共享模块、数据依赖） | 统一设计一套架构，所有 Spec 的需求一次覆盖 |
   | **弱相关**（少量接口交互） | 各自独立设计，但接口定义时对齐，标注跨 Spec 边界 |
   | **不相关**（独立功能域） | 完全独立设计，互不影响 |

2. 如果用户提供了技术栈偏好，优先使用
3. 标识可扩展点

**Phase 3：方案输出**
1. 根据关系类型选择输出形式：统一架构 / 标注边界 / 分开输出
2. 输出架构设计方案

**输出**：

```markdown
# 架构设计方案

## 多 Spec 关系统计
- 本次覆盖 Spec 数量：N 个
- 关系类型：强相关 / 弱相关 / 不相关
- 共享模块数量：N 个

## 技术选型
- 语言/框架：...
- 关键库：...
- 选型理由：...

## 模块划分
| 模块 | 职责 | 所属 Spec | 对外接口 | 新增/修改 |
|------|------|----------|---------|----------|
| ...  | ...  | Spec-01   | ...      | 新增      |

| Spec | 功能域 | 涉及模块 |
|------|-------|---------|
| Spec-01 | 语音唤醒 | 唤醒引擎、指令解析 |

## 数据流
- 请求流程：...
- 关键路径：...

## 接口定义
- API / 方法签名：...

## 架构影响分析
### 与已有架构的关系
- 新增模块 vs 已有模块
- 对已有模块的影响范围
- 是否需要修改已有模块

### 架构一致性检查
| 维度 | 检查结果 | 说明 |
|------|---------|------|
| 模块职责 | ✅ / ⚠️ / ❌ | 新模块与已有模块无职责重叠 |
| 接口风格 | ✅ / ⚠️ / ❌ | 新 API 与项目风格一致 |
| 技术栈 | ✅ / ⚠️ / ❌ | 无冲突的技术选型 |
| 数据流 | ✅ / ⚠️ / ❌ | 数据流集成合理 |

### 现有架构优化建议（如有）
- 检测到现有模块 X 的职责过重，建议拆分
- 检测到模块 Y 存在硬编码依赖，建议改为接口注入

## 架构边界与演进方向
### 当前覆盖范围
- 本次设计覆盖的功能域：...

### 可扩展点（标记位置，不做预实现）
- 接口 `XXX` 设计了通用格式，未来新增类型无需修改核心逻辑
- 模块 `YYY` 预留了插件位，当前只实现默认策略

### 领域演进提示（如有）
- 检测到项目所属领域（如车载语音），常见演进方向包括...
- 当前设计已兼容上述方向，但未实际实现
```

**输出方式根据关系类型不同**：

| 关系 | 输出方式 |
|------|---------|
| **强相关**（共享模块、数据依赖） | 一套统一架构方案，包含需求→模块映射表 |
| **弱相关**（少量接口交互） | 统一架构，标注模块边界和跨 Spec 接口 |
| **不相关**（独立功能域） | 分开的多套架构方案，各出一份，在同一个消息中返回 |

**硬性规则**：
- **有项目上下文时**：必须做一致性检查，不一致项不得跳过，必须在输出中标注
- **无项目上下文时**：必须提示用户"未检测到项目上下文"
- **多个 Spec 时**：必须先独立设计再交叉分析，不得跳过 Phase 1 直接合并
- **不相关 Spec**：必须分开输出，不得强行合并为统一架构
- 必须显式标注"新增、修改、不变"三类模块的关系
- 不设计实现细节（留给代码生成阶段）
- 如果检测到现有架构有明显不足，必须提出优化建议（不强制采纳，但需要标识）
- **不做未来需求的预实现** — 可扩展点只标记接口位和兼容性说明，不写入具体实现逻辑

> **输出提示**：建议将架构设计方案保存为文件，作为项目架构文档供后续需求参考。

### 5.3 task-planner（任务规划）

**职责**：将架构设计方案拆解为可独立开发的任务列表。

**触发条件**：
- 用户要求"拆任务"
- 用户说"/ml.plan"

**输入**：
- 架构设计方案（来自用户粘贴、文件引用或对话上下文）
- 可选的排期/资源约束

**输出**：

```markdown
# 任务列表

## 任务清单
| ID | 任务 | 所属模块 | 依赖 | 预估工作量 | 验收条件 |
|----|------|---------|------|-----------|---------|
| T-01 | ... | ... | ... | ... | ... |

## 执行顺序
- T-01 → T-02 → T-03（串行依赖）
- T-04、T-05 可并行执行
```

**处理流程**：
1. 基于模块划分拆解任务
2. 识别任务间的依赖关系
3. 标记可并行执行的任务
4. 为每个任务指定验收条件

> **输出提示**：建议将任务列表保存为文件，以接入项目管理工具或供后续会话使用。

## 6. Skill 间协作机制

MuniuLiuma 的 Skill 之间**不依赖任何内置的编排机制**。协作完全由 AI Agent 和用户驱动：

1. **定义清晰的输入输出** — 每个 Skill 的输入要求和输出格式在 SKILL.md 中明确定义
2. **Agent 自行编排** — Agent 根据用户意图（如"解析这个 PRD 再做架构设计"）判断需要调用的 Skill 和顺序
3. **对话上下文传递数据** — 前一个 Skill 的输出保留在对话中，Agent 将其作为下一个 Skill 的输入传递
4. **用户介入** — Agent 发现输入不足时主动询问用户，用户补充后继续

**数据依赖关系**（仅作参考，不强制执行顺序）：

```
prd-parser 输出               arch-designer 输出
┌────────────────────┐        ┌────────────────────┐
│ 结构化 Spec         │ ───→  │ 架构设计方案        │ ───→ ...
│   - 功能点清单      │        │   - 模块划分        │
│   - 验收标准        │        │   - 接口定义        │
│   - 约束条件        │        └────────────────────┘
│   - 模糊项          │               │
│   - 依赖关系        │               ▼
└────────────────────┘        ┌────────────────────┐
         │                    │ task-planner 输出    │
         ▼                    │   - 任务清单         │
┌────────────────────┐        └────────────────────┘
│ test-planner 输出   │
│   - 测试矩阵        │
│   - 用例模板        │
└────────────────────┘
```

> **各 Skill 可独立使用**。例如用户可以直接调用 `/ml.design` 并粘贴 Spec，无需先执行 prd-parser。Agent 根据用户的具体要求决定调用路径。

### 7.2 每个 SKILL.md 的格式

name: "muniu-prd-parser"
