# Phase 1：prd-parser（需求解析）

## 职责

将原始 PRD / 需求文档 / 功能描述转化为结构化 Spec，确保不遗漏任何需求细节、不放过任何模糊项。

## 输入

- 纯文本（粘贴或消息中的文字）
- 本地文件内容（读取后传入，如 `.md` / `.txt`）
- 网页内容（通过 WebFetch 抓取后传入）
- 无输入时主动追问获取信息
- **多份文档**：逐份处理，每份产出对应的独立 Spec

> **只接受文本输入**。文件读取、网页抓取、内容提取均在调用前完成。

## 处理流程：三遍校验法

### 第一遍：粗提取（广度覆盖）

通读全文，识别所有功能点和约束条件。

- 宁可多列不漏列
- 提取所有显式和隐式的功能需求
- 标记约束条件（技术约束、业务约束、安全约束）
- 输出：功能点清单 + 约束条件草稿

### 第二遍：细校验（精度保证）

逐段对照原文，检查第一遍的提取结果。

- 逐段扫描原始文档，对比已提取的功能点
- 补充遗漏的功能点、约束条件、业务规则
- 修正第一遍中理解偏差的条目
- **第二遍未完成前，不得输出最终结果**

### 为每个功能点生成验收标准

- 格式：Given / When / Then
- 每个功能点至少一个验收标准
- 验收标准必须是可客观验证的描述

### 第三遍：模糊检测（质量达标）

逐条检查每个功能点的表述是否明确。按以下 8 个维度检测：

| 模糊类型 | 检测方式 | 追问示例 |
|---------|---------|---------|
| **量化缺失** | 无具体数字/指标 | "快速"的具体指标是？<1s？<500ms？ |
| **主语缺失** | 谁来做？ | 前端校验还是后端校验？ |
| **条件缺失** | 什么条件？什么结果？ | 具体异常类型？提示文案/弹窗？ |
| **程度模糊** | "主流""常见"等无定义词 | "主流车型"具体指哪些？ |
| **范围模糊** | "部分""某些"无划分规则 | 划分规则是什么？按角色/标签/权限？ |
| **时态模糊** | "后续""将来"无明确版本 | 当前版本做不做？P0 还是未来规划？ |
| **参照模糊** | "类似 XX 那样"无具体指向 | 微信的哪个具体功能？交互/视觉/全部？ |
| **冲突未处理** | 多处说法不一致 | A 处说"必须登录"，B 处说"游客可用"，以哪个为准？ |

#### 矛盾检测（v0.3 新增）

除了单条模糊表述，还需检测需求间的交叉矛盾：

| 矛盾类型 | 检测方式 | 追问示例 |
|---------|---------|----------|
| **功能矛盾** | 交叉对比功能点要求，检查是否有冲突 | A 要求"实时同步"，B 要求"批量处理"，以哪个为准？ |
| **约束矛盾** | 检查约束条件间是否冲突 | 技术约束要求"轻量级"，业务约束要求"支持并发1000"，是否兼容？ |
| **优先级矛盾** | 同一功能在不同段落的优先级是否一致 | 第3章说 P0，第5章说 P2，以哪个为准？ |
| **时序矛盾** | 检查依赖关系是否可行（循环依赖、不可能序列） | A 依赖 B，B 依赖 C，C 依赖 A，依赖循环如何打破？ |

矛盾检测结果纳入模糊项表格，标注矛盾类型和影响范围（涉及哪些功能点）。

- **第三遍检测出模糊项但未生成追问，不得输出最终结果**

### 分析依赖关系

- 识别功能间的依赖关系（A 依赖 B 才能运行）
- 识别功能与已有系统的依赖

## 输出格式

```markdown
# 结构化 Spec：[文档标题]

## 功能点清单
| ID | 功能点 | 优先级 | 关联需求 |
|----|-------|--------|---------|
| F-01 | ... | P0 | ... |

## 验收标准
| ID | 功能点 | Given | When | Then |
|----|-------|-------|------|------|
| AC-01 | F-01 | ... | ... | ... |

## 约束条件
- 技术约束：...
- 业务约束：...
- 安全约束：...

## 模糊项
| 问题 | 建议 | 优先级 |
|------|------|--------|
| "..."未明确 | 建议确认... | 高/中/低 |

## 依赖关系
- 本功能依赖已有功能：...
- 本功能被其他功能依赖：...
```

## 上下文验证标记

每个 Phase 1 输出末尾附加验证标记，供下游 Phase 检查上游产物完整性：

```markdown
<!-- CONTEXT-CHECK: F-{功能点数量}, AC-{验收标准数量}, C-{约束条件数量}, Q-{模糊项数量} -->
```

下游 Phase 开始前，检查上游产物的验证标记是否存在且数值合理。如缺失或异常，提示用户"上游产物可能不完整，建议确认后继续"。

## 硬性规则

- 每个功能点**必须**有至少一个验收标准
- 模糊项**必须**标记，不能默认"理解" — 宁可多问不错过
- **必须**区分 P0 / P1 / P2 优先级
- 不修改用户原始意图，只做结构化提取
- **第二遍校验未完成前，不得输出最终结果**
- **第三遍检测出模糊项但未生成追问，不得输出最终结果**
- 多份文档时，每份独立输出，不合并

## 追问回填机制

当用户回答追问清单中的问题后，将答案回填到 Spec：

1. 将答案回填到 Spec 对应的功能点描述中
2. 更新受影响的验收标准
3. 将已回填的模糊项状态从"待确认"更新为"已确认"
4. 如追问答案引发新的模糊项 → 补充到追问清单

回填完成后输出更新摘要："已回填 N 项，剩余 M 项待确认，新增 K 项追问。"

> 无需重新执行完整的 prd-parser 流程，直接在已有 Spec 上更新。

## 增量更新场景

当用户提供已解析过的 PRD 的更新版本时：

1. 对比新旧版本 PRD 的差异
2. 仅重新解析变更部分，更新 Spec 中对应的功能点、验收标准和约束条件
3. 保留未变更部分的原有 Spec 内容
4. 标注变更影响范围："以下 N 个功能点因 PRD 更新而变更，下游产物可能已过期"

### Delta 分类（v0.3 新增）

增量更新时，每个功能点变更必须标注 Delta 类型：

- **ADDED**：新增功能点（包含完整功能点描述、验收标准）
- **MODIFIED**：修改已有功能点（包含变更说明 + 修改前文本 + 修改后文本）
- **REMOVED**：删除已有功能点（标注删除原因）

### 增量 Spec 输出格式

```markdown
# 增量 Spec 更新

## Delta 统计
- ADDED: N 个功能点
- MODIFIED: N 个功能点
- REMOVED: N 个功能点

## 变更详情

### ADDED
| ID | 功能点 | 优先级 | 关联需求 |
|----|-------|--------|----------|
| F-XX | ... | P1 | ... |

### MODIFIED
| ID | 功能点 | 变更说明 | 修改前 | 修改后 |
|----|-------|---------|-------|-------|
| F-XX | ... | 新增了XX条件 | 原文 | 新文 |

### REMOVED
| ID | 功能点 | 删除原因 |
|----|-------|----------|
```

> 增量更新仅在用户明确提供"更新版 PRD"时触发，普通新 PRD 仍走全量解析。
