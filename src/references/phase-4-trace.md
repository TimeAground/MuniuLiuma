# Phase 4：trace（完工审计）

## 职责

汇聚开发全流程全部产物，做完工审计，判断需求是否全部完成，标识遗漏或未达标的交付缺口。

## 输入

- **[必需] 结构化 Spec** — 功能点清单、验收条件（prd-parser 输出）
- **[必需] 架构设计方案** — 模块划分（arch-designer 输出）
- **[必需] 任务清单** — 任务 ID 和依赖关系（task-planner 输出）
- **[必需] 实现代码** — 项目代码（用于代码扫描和审计）

**前置依赖处理**：部分输入缺失时按以下优先级补齐：

1. **检查对话上下文** — 优先从当前对话中查找已有产物
2. **缺失时自动补齐** — 逐层回溯调用上游 Phase
3. **用户提供** — 如补齐链路不可行，询问用户补充
4. **部分审计** — 如用户明确要求"只看能查的部分"，跳过缺失维度，标注"XX 维度因缺少 XX 输入未审计"

## 处理流程

### 1. 收集输入

从对话上下文或用户提供方式获取全链路产物，按前置依赖规则补齐。

### 2. 执行审计：五个维度

| 审计维度 | 检查内容 | 数据来源 |
|---------|---------|---------|
| **功能覆盖** | Spec 中每个功能点是否有对应的任务记录或代码实现 | Spec × 任务清单 × 代码 |
| **架构落地** | 架构设计中定义的每个模块是否在代码中有对应实现 | 架构设计 × 代码 |
| **无主代码** | 是否有实现了但不对应任何 Spec 功能点的代码 | 代码 × Spec（反向扫描） |
| **任务闭环** | 全部任务是否在代码中有对应实现 | 任务清单 × 代码 |
| **验收标准达标** | Spec 中 Given/When/Then 场景是否在代码逻辑中被满足 | Spec 验收标准 × 代码逻辑 |

### 3. 代码扫描判定标准

| 判定方式 | 适用场景 | 说明 |
|---------|---------|------|
| 文件/目录存在性检查 | 模块级审计 | 检查架构定义的模块是否有对应目录/文件 |
| 函数/类名匹配 | 任务级审计 | 精确匹配 + 模糊匹配（语义相近） |
| 调用链追踪 | 功能覆盖审计 | 从入口点追踪完整功能路径 |

**语言适配**：
- 动态语言（Python/JS/TS）：侧重运行时调用分析和 import/require 分析
- 静态语言（Java/Go/Rust）：利用类型系统和编译时检查

**置信度标注**：每个判定结果标注置信度：
- ✅ **确认** — 高置信度匹配
- ⚠️ **疑似** — 模糊匹配，需用户确认
- ❌ **未实现** — 未找到对应代码

### 4. 判定交付状态

- ✅ **可交付**：全部功能点已实现，架构模块已落地，无无主代码
- ⚠️ **有条件交付**：P0 全部实现，低优先级有缺口
- ❌ **不可交付**：存在 P0 未实现

## 硬性规则

- **五个维度全部覆盖**，不得跳过任何维度（除非输入缺失并明确标注）
- 每个判定结果**必须**标注置信度（✅ 确认 / ⚠️ 疑似 / ❌ 未实现）
- P0 未实现**必须**判定为"❌ 不可交付"
- 无主代码**必须**在报告中标注，不得忽略
- 验收标准验证**必须**逐条对照 Spec 中的 Given/When/Then
- 降级审计**必须**标注哪些维度被跳过及原因
