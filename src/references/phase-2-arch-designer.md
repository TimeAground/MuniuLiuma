# Phase 2：arch-designer（架构设计）

## 职责

基于结构化 Spec 和项目现有架构上下文，输出一致、可落地的架构设计方案。聚焦当前需求，同时标识可扩展点以备未来演进。

## 输入

- **[必需] 结构化 Spec** — 来自 prd-parser 输出，或用户直接提供。接受单个或多个 Spec。
- **[可选] 项目现有架构上下文** — 在项目目录中自动收集：
  - 技术栈（语言、框架、数据库、中间件）
  - 现有模块划分及职责
  - API / 接口风格
  - 关键数据流和依赖关系
  - 已有架构文档（如有）
- 用户可补充技术栈偏好

> 无项目上下文时正常设计，输出末尾提示："未检测到项目上下文，建议在项目目录中调用以获得更好的架构一致性。"

**前置依赖**：不允许在无 Spec 的情况下进行推测性设计。Spec 缺失时，先检查有无原始 PRD → 有则先跑 Phase 1 → 无则询问用户。

## 项目上下文收集范围

| 收集项 | 来源示例 | 收集深度 |
|---------|---------|----------|
| 技术栈 | `package.json`、`pom.xml`、`go.mod` | 直接读取配置文件 |
| 模块划分 | 项目目录结构、`src/` 子目录 | 扫描目录层级（最深 3 层） |
| 现有 API/接口 | 路由文件、Controller 类、OpenAPI | 读取路由定义，不深入实现 |
| 已有架构文档 | `ARCHITECTURE.md`、`docs/` | 仅读取文档摘要 |
| 依赖关系 | 模块间 import/引用 | 分析顶层依赖，不做递归分析 |

> 优先读取配置文件和目录结构（成本低、信息密度高），代码分析仅作补充。
> 项目规模较大时，可主动询问用户"项目核心模块有哪些"以替代全量扫描。

## 处理流程

### 步骤 A：交叉分析

1. 读入所有 Spec，识别 Spec 数量和功能域
2. 分析 Spec 间的关系：
   - **模块重叠检测** — 是否有多个 Spec 定义了相同或相似的模块
   - **接口依赖检测** — 是否有 Spec 的输出是另一个 Spec 的输入
   - **技术栈冲突检测** — 是否有 Spec 提出了不一致的技术选型
   - **完全不相关** — 多个 Spec 分属不同的功能域
3. 如有项目上下文，分析新设计对现有架构的影响
4. 输出关系类型：强相关 / 弱相关 / 不相关

### 步骤 B：分层设计

根据关系类型选择设计策略：

| 关系 | 策略 |
|------|------|
| **强相关** | 统一设计一套架构，所有 Spec 的需求一次覆盖 |
| **弱相关** | 各自独立设计，接口定义时对齐，标注跨 Spec 边界 |
| **不相关** | 完全独立设计，互不影响 |

- 如用户提供了技术栈偏好，优先使用
- 标识可扩展点（只标记位置，不做预实现）

### 步骤 C：风险评估

评估以下维度的技术风险：

- **技术风险**：新技术/新库的成熟度、稳定性、社区活跃度
- **实现风险**：方案复杂度是否超出团队经验
- **性能风险**：关键性能指标是否可达成
- **集成风险**：与现有系统的兼容性问题

### 步骤 D：方案输出

根据关系类型选择输出形式：统一架构 / 标注边界 / 分开输出。

## 硬性规则

- **有项目上下文时**：必须做一致性检查，不一致项必须在输出中标注
- **无项目上下文时**：必须提示"未检测到项目上下文"
- **多个 Spec 时**：必须先独立设计再交叉分析，不得跳过步骤 A 直接合并
- **不相关 Spec**：必须分开输出，不得强行合并为统一架构
- 必须显式标注"新增、修改、不变"三类模块
- 不设计实现细节（留给代码生成阶段）
- 如检测到现有架构有明显不足，必须提出优化建议
- **不做未来需求的预实现** — 可扩展点只标记接口位和兼容性说明
- **有项目上下文时**：必须评估技术风险，高风险项不得跳过
