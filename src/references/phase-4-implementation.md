# Phase 4：implementation（实现指导）

## 职责

将任务清单转化为结构化的实现指导规范，为 Agent（或人类开发者）提供编码阶段的行为约束、执行策略和质量基准。

## 输入

- **[必需] 任务清单** — 任务 ID、依赖关系、验收条件、证据要求（task-planner 输出）
- **[必需] 架构设计方案** — 模块划分、接口定义、技术选型（arch-designer 输出）
- **[可选] 结构化 Spec** — 功能点、验收标准、约束条件（用于生成一致性测试要求）

**前置依赖处理**：不允许在无任务清单的情况下生成实现指导。任务清单缺失时按以下优先级补齐：

1. 检查上下文有无架构设计方案 → 有则先调用 task-planner
2. 检查上下文有无结构化 Spec → 有则先调用 arch-designer
3. 均缺失 → 继续回溯至 Phase 1

## 处理流程

### 步骤 A：上下文完整性检查

检查上游产物是否包含验证标记：

```markdown
<!-- CONTEXT-CHECK: F-{功能点数量}, AC-{验收标准数量}, C-{约束条件数量}, Q-{模糊项数量} -->
```

- 如验证标记存在且数值合理 → 继续生成
- 如验证标记缺失或数值异常 → 提示用户"上游产物可能不完整，建议确认后继续"

### 步骤 B：生成三层边界约束

从架构设计的约束条件和 Spec 的约束条件，派生 Agent 行为边界：

| 层级 | 含义 | 生成规则 |
|------|------|---------|
| ✅ Always | 无需询问即可做 | 运行测试、遵循命名规范、记录日志、使用已有依赖 |
| ⚠️ Ask First | 需人类确认 | 修改数据库 Schema、添加新依赖、变更 API 签名、删除文件/目录、修改配置 |
| 🚫 Never | 绝对禁止 | 提交密钥/凭证、修改 CI 配置、删除测试用例、绕过安全检查 |

**生成原则**：
- 从架构设计的约束条件中直接提取硬性约束（如"不得引入新框架"→ 🚫 Never）
- 从 Spec 约束条件中提取业务约束（如"必须支持多租户"→ ✅ Always 在每次修改时检查多租户兼容性）
- 通用安全规则始终包含在 🚫 Never 中（提交密钥、删除测试等）

### 步骤 C：生成分阶段执行策略

将任务按开发阶段分组，形成阶段化的执行计划：

**阶段划分规则**：
1. **Setup 阶段** — 基础设施搭建（项目初始化、依赖安装、配置、数据库 Schema）
2. **Core 阶段** — 核心功能实现（P0 功能点的主路径）
3. **Advanced 阶段** — 增强功能（P1/P2 功能、错误处理、边界情况）
4. **Testing 阶段** — 测试与验证（单元测试、集成测试、性能测试）

**执行策略**：
- 阶段间**串行**：前一阶段审查门禁通过后，才进入下一阶段
- 阶段内**并行**：无依赖关系的任务可并行执行
- 每个审查门禁检查：
  - 阶段内任务是否全部完成
  - 验收条件是否达标
  - 证据是否提供（如任务清单有证据要求）
  - 是否有需要回退到上游 Phase 的问题

### 步骤 D：生成一致性测试要求

从 Spec 和架构设计中派生测试要求（语言无关）：

**派生规则**：
- Spec 验收标准（Given/When/Then） → 功能测试断言
- 架构设计 API 定义（输入/输出/状态码） → API 测试用例
- Spec 约束条件（性能指标） → 基准测试要求
- Spec 约束条件（安全要求） → 安全检查项
- 架构设计数据模型 → 数据完整性验证

**测试类型标注**：每条测试要求标注类型（功能/性能/安全/数据完整性）

### 步骤 E：组装输出

将三层边界约束、分阶段执行策略、一致性测试要求组装为实现指导规范。

## 输出格式

```markdown

| 层级 | 规则 | 来源 |
|------|------|------|
| ✅ Always | ... | 架构约束 C-XX / 通用规则 |
| ⚠️ Ask First | ... | Spec 约束 C-XX |
| 🚫 Never | ... | 安全规则 |
```

## 硬性规则

- **三层边界约束必须生成**，不得跳过任何层级（即使某层级规则较少）
- **分阶段执行策略必须覆盖全部任务**，每个任务归属且仅归属一个阶段
- **一致性测试要求必须逐条派生**，不得从 Spec 验收标准中遗漏
- 任务间的依赖关系**必须**在阶段划分中保留（如 T-03 依赖 T-01，T-01 必须在 T-03 所在阶段之前）
- 审查门禁**必须**包含阶段内全部任务的验收条件检查
- 🚫 Never 层级**必须**包含安全相关的硬性禁止规则
- 输出语言遵循 SKILL.md 语言策略（默认中文）
