# Phase 3：task-planner（任务规划）

## 职责

将架构设计方案拆解为可独立开发的任务列表。

## 输入

- **[必需] 架构设计方案** — 来自 arch-designer 输出（模块划分、接口定义），或用户直接提供的设计描述

**前置依赖**：不允许跳过架构设计直接生成任务。架构设计缺失时，先检查有无 Spec → 有则先跑 Phase 2 → 无则继续回溯至 Phase 1。

## 处理流程

### 1. 识别架构层次

按以下层次逐层提取模块：

```
UI 层 → 业务逻辑层 → 数据访问层 → 基础设施层
```

层内按组件/功能进一步细分。

### 2. 按拆分规则生成任务

**拆分依据（按优先级）**：

- **按架构层次**：每一层作为一个任务分组，层内各组件独立成任务
- **按接口/数据流**：每个 API 端点、每条数据流处理路径独立为一个任务
- **按功能点**：一个完整功能的最小可验证单元独立为任务

**粒度标准**：

- 理想任务规模：独立可开发、可测试、1-2 小时完成，最长不超过 4 小时
- 超 4 小时的任务须进一步拆解
- 不足 30 分钟的任务合并到相邻任务中

**依赖识别规则**：

- 被依赖模块（数据模型、工具函数、基类等）优先拆出，排入早期任务
- 同一层次内无相互引用的组件标记为可并行执行
- 依赖链长度不超过 3 层（A → B → C），超过则合并中间任务

**工作量估算标准**：

| 级别 | 时间 | 典型场景 |
|------|------|---------|
| 简单 | ≤1h | 修改变量名、配置文件、单一工具函数 |
| 中等 | 1-2h | 新增一个 CRUD 接口、一个无交互的 UI 组件 |
| 复杂 | 2-4h | 新增一个完整页面含交互、跨层联调功能 |

**验收条件要求**：

- 必须为可客观验证的描述
- ✅ "用户注册接口返回 201 且密码已加密存储"
- ❌ "注册功能正常"

### 3. 识别依赖关系

根据依赖规则推断依赖图，标记串行和并行关系。

### 4. 标记可并行任务

同一层次无依赖关系的任务归入同一并行批次。

### 5. 为每个任务指定验收条件

按验收条件要求编写可验证断言。

## 输出格式

```markdown
# 任务列表

## 任务清单
| ID | 任务 | 所属模块 | 依赖 | 预估工作量 | 验收条件 |
|----|------|---------|------|-----------|---------|
| T-01 | 实现用户注册 API | 用户服务 | — | 中等（2h） | POST /register 返回 201，密码已加密 |
| T-02 | 实现用户登录 API | 用户服务 | T-01 | 中等（1.5h） | POST /login 返回 JWT Token |

## 执行顺序
- 串行依赖：T-01 → T-02 → T-03
- 可并行：T-01 和 T-04 可并行开发
```

## 任务状态跟踪

不管理任务状态，但输出格式支持状态追踪：

- 每个任务 ID 可附加状态标记：`待开发` / `开发中` / `已完成` / `已验证`
- 用户再次调用 `/mnlm.plan` 时，如上下文中已有任务状态信息，仅更新未完成的任务，已完成的保留原样
- 任务清单可导出为 Markdown 表格或简单 CSV

## 任务变更与回退

当某个任务在开发中发现不可行时，可请求重新规划：

1. 标记失败任务及原因
2. 生成替代任务或调整方案
3. 重新评估受影响任务的依赖链和执行顺序
4. 标注变更原因和影响范围："任务 T-XX 因 XX 原因调整，影响下游任务 T-YY、T-ZZ"

## 硬性规则

- 每个任务**必须**有明确的验收条件（可客观验证）
- 超 4 小时的任务**必须**进一步拆解
- 不足 30 分钟的任务**必须**合并
- 依赖链长度**不得超过 3 层**
- **不允许**跳过架构设计直接生成任务
- 被依赖模块**必须**排入早期任务
